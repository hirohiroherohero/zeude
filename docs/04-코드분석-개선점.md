# Zeude 코드 분석: 개선 가능한 점

> 만약 우리가 직접 만들거나, Fork 후 커스터마이징한다면 고려할 점들

---

## 1. 아키텍처

### ClickHouse는 소규모 팀에 과하다

| 현재 | 개선안 |
|------|--------|
| Supabase(설정) + ClickHouse(분석) | Supabase 하나로 통합 (TimescaleDB 확장 활용) |
| 3개 시스템 운영 필요 | 1개로 줄여서 진입장벽 낮춤 |
| 하루 수백만 건 이상일 때 의미 | 50명 이하 팀은 PostgreSQL로 충분 |

→ ClickHouse는 규모가 커지면 그때 추가하는 것이 합리적

### CLI Shim이 Go로 되어 있어 빌드 부담

- 현재: Go로 작성 → OS/아키텍처별 크로스 컴파일 필요
- 개선안: TypeScript/Node.js로 작성하면 컴파일 불필요
- 단, Go의 장점(빠른 시작, 낮은 메모리)은 존재하므로 트레이드오프

## 2. 코드 품질 이슈

### 버전 비교 로직 버그 (심각도: 높음)

```go
// zeude/internal/autoupdate/autoupdate.go
// 문자열 비교로 버전을 판단 → "1.9" > "1.10" 이 된다 (오류)
if remoteParts[i] > localParts[i] { return true }
```

현재는 `YYYYMMDD.HHMM` 포맷이라 우연히 동작하지만, semver에서는 깨짐.

### Rate Limiting이 인메모리 (심각도: 높음)

```typescript
// zeude/dashboard/src/lib/rate-limit.ts
const store = new Map<string, RateLimitEntry>()  // 서버 메모리에만 존재
```

서버 재시작 시 초기화, 수평 확장 시 인스턴스마다 별도 카운트.
→ Redis 또는 Upstash 기반 분산 Rate Limiting 필요

### 핵심 모듈 테스트 부재 (심각도: 높음)

- `internal/mcpconfig/sync.go` (82KB) — CLI 핵심 동기화 로직, 테스트 없음
- 훅 실행 로직 — 테스트 없음
- 대시보드 컴포넌트 — 테스트 없음

### 개발 모드 인증 우회 (심각도: 중간)

```typescript
// SKIP_AUTH=true 설정 시 모든 요청이 admin으로 처리됨
if (process.env.NODE_ENV === 'development' && process.env.SKIP_AUTH === 'true') {
  return { ...mockSession, role: 'admin' }
}
```

## 3. 누락된 기능

| 항목 | 설명 |
|------|------|
| **바이너리 롤백** | 자동 업데이트 실패 시 복구 메커니즘 없음 |
| **MCP 명령어 검증** | 등록한 명령어의 유효성/보안 검증 없음 |
| **스킬 미리보기** | 유저 시점 미리보기 불가 |
| **동기화 실패 가시성** | CLI 동기화 실패 시 대시보드에서 확인 불가 |
| **스킬 추천 퍼지매칭** | 정확 일치만 지원, 유사어 매칭 안 됨 |

## 4. 만약 처음부터 만든다면?

| 구분 | 현재 Zeude | 권장 스택 |
|------|-----------|----------|
| Frontend | Next.js | Next.js (유지) |
| Backend | Next.js API Routes | Next.js API Routes (유지) |
| DB (설정) | Supabase | Supabase (유지) |
| DB (분석) | ClickHouse | Supabase + TimescaleDB (초기) |
| CLI Shim | Go | TypeScript (빌드 부담 제거) |
| Auth | 자체 OTT | Supabase Auth (이미 내장) |
| Rate Limit | 인메모리 Map | Upstash Redis |
