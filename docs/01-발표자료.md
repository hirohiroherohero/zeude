# Zeude 도입 검토 발표자료

## 1. Zeude란?

Zeude는 **Claude Code의 전사 도입 및 모니터링을 위한 오픈소스 플랫폼**이다. ZEP에서 개발했으며 Apache 2.0 라이선스로 공개되어 있다.

핵심 철학: **"Measurement brings visibility, sharing drives adoption."**
(측정이 가시성을 만들고, 공유가 도입을 이끈다)

---

## 2. 왜 필요한가? (도입 배경)

### 현재 우리가 겪고 있는 문제

| 문제 | 설명 |
|------|------|
| **낮은 활용률** | Claude Code 라이선스를 구매했지만 실제 활용하는 개발자가 제한적 |
| **노하우 사일로** | 잘 쓰는 사람의 프롬프트, 스킬, 워크플로우가 개인에게만 남아있음 |
| **사용 현황 파악 불가** | 누가, 얼마나, 어떻게 쓰고 있는지 데이터가 없음 |
| **설정 관리 부재** | MCP 서버, 훅, 스킬 등을 개발자 각자가 수동으로 설정해야 함 |

### Zeude가 제시하는 해결 구조 (3 Layer)

```
Layer 1: Sensing (측정)     → OpenTelemetry로 사용 데이터 자동 수집
Layer 2: Delivery (배포)    → 스킬, MCP, 훅을 중앙에서 전사 배포
Layer 3: Guidance (가이드)  → 프롬프트에 맞는 스킬을 실시간 추천
```

### 실제 사례

ZEP의 실 배포 사례에서 **별도의 강제 없이 데이터 기반 접근만으로 Claude Code 활용률이 6% → 18%로 3배 증가**했다고 보고되어 있음.

## 3. 주요 기능

### 대시보드 (Dashboard)
- **Overview**: 전사 Claude Code 사용 현황 한눈에 파악
- **Sessions**: 개별 세션 이력 조회
- **Daily Stats**: 일별 사용 통계
- **Leaderboard**: 활용도 높은 개발자 순위 (자연스러운 경쟁 유도)

### 워크스페이스 관리
- **Skills**: 슬래시 커맨드(`/skill-name`)로 프롬프트 템플릿을 팀 전체에 공유
- **Hooks**: Claude Code 이벤트(프롬프트 입력, 도구 호출 등)에 자동 스크립트 실행
- **MCP Servers**: 외부 도구/API 연결을 중앙에서 일괄 배포
- **Agents**: Claude의 역할/성격을 커스텀 프로필로 정의하여 배포

### 팀 관리
- **Team Members**: 초대 링크 기반 온보딩, 역할(admin/member) 관리
- **Analytics**: 팀별 사용 분석 (ClickHouse 필요)

### 자동 동기화
- Admin이 대시보드에서 설정하면 팀원 CLI에 자동 반영 (5분 내)
- 개발자는 평소처럼 `claude`만 쓰면 됨

## 4. 기술 스택 및 데이터 구조

### 기술 스택

| 구성요소 | 기술 | 역할 |
|---------|------|------|
| CLI (Shim) | Go | claude 명령어 래핑 + 텔레메트리 전송 |
| Dashboard | Next.js (React) | 관리 UI + API |
| DB (설정) | Supabase (PostgreSQL) | 유저, 설정, 스킬 저장 |
| DB (분석) | ClickHouse | 대용량 텔레메트리 저장/분석 |
| Telemetry | Vercel API Routes (OTel proxy) | 표준 텔레메트리 수집 파이프라인 |

### 왜 DB를 두 개 쓰는가?

| 구분 | Supabase (PostgreSQL) | ClickHouse |
|------|----------------------|------------|
| **용도** | 설정, 상태, 관계형 데이터 | 시계열 분석, 대용량 로그 |
| **데이터 크기** | 작음 (유저/설정) | 큼 (하루 수백만 건 로그) |
| **쿼리 패턴** | 단건 조회/수정 | 대량 집계 (SUM, AVG, GROUP BY) |
| **보존 기간** | 영구 | TTL 기반 자동 삭제 (90~180일) |

### 데이터 흐름

```
개발자가 claude 실행
    │
    ├─ [설정 조회] ──→ Supabase (스킬, 훅, MCP 가져옴)
    │
    └─ [텔레메트리] ──→ /api/otel/* (Vercel) ──→ ClickHouse
                                            ├─ claude_code_logs (원본)
                                            ├─ token_usage_hourly (집계)
                                            └─ ai_prompts (프롬프트)
                                                    │
                                        Dashboard에서 분석/시각화
```

## 5. 시연 (라이브 데모)

**대시보드 URL:** https://zeude-vwuo.vercel.app

### 시연 항목
1. Skills — 스킬 생성 및 `/skill-name`으로 호출
2. Hooks — 이벤트 트리거 스크립트 등록
3. MCP Servers — 외부 도구 연결
4. Agents — AI 역할 프로필 배포
5. Team — 멤버 초대 및 관리

## 6. 도입 로드맵

### Phase 1: PoC (승인 후 1~2주, 비용 0원)

```
1. 회사 GitHub org에 원본 레포 Fork
2. Vercel(무료) + Supabase(무료)로 대시보드 배포
3. 팀원 5~10명 초대 → 스킬/훅/MCP 기능 검증
```

### Phase 2: 파일럿 확대

- ClickHouse 추가 → 텔레메트리/분석 활성화
- 20~30명으로 확대, 사용 데이터 기반 효과 측정

### Phase 3: 전사 도입

- 정식 인프라 구축 (사내 서버 또는 클라우드)
- 전 개발팀 온보딩

---

## 7. 기대 효과 및 리스크

### 기대 효과
- Claude Code **사용 현황의 정량적 파악** 가능
- 팀 간 **베스트 프랙티스 공유** 체계 구축
- MCP, 훅, 스킬의 **중앙 관리로 설정 부담 제거**
- 데이터 기반의 **자연스러운 활용률 향상** (강제 아닌 유도)

### 리스크
- 오픈소스 초기 단계로 문서화/안정성이 완벽하지 않음
- 텔레메트리 수집에 대한 개발자 동의/프라이버시 정책 필요
- ClickHouse 인프라 운영 비용 발생 (Phase 2부터)
